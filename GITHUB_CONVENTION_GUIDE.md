# 팀 프로젝트 GitHub 커밋 및 PR 컨벤션 가이드

팀원 여러분, 원활한 협업과 코드 관리를 위해 GitHub 커밋 및 Pull Request(PR) 작성 규칙을 아래와 같이 안내해 드립니다. 코드를 업로드하실 때 반드시 이 양식을 지켜주시기 바랍니다.

## 1. 커밋 메시지 컨벤션 (Commit Convention)

커밋 메시지는 항상 아래의 형식을 지켜주세요.

**기본 형식:**
```
카테고리: 수정내용 간략하게
```
*(주의: 카테고리와 수정내용 사이에 콜론(`:`)과 띄어쓰기를 지켜주세요)*

### 📌 사용 가능한 카테고리 목록

*   **`feat`**: 새로운 기능 추가
*   **`fix`**: 버그 수정
*   **`refactor`**: 코드 구조 개선 (기능 변경 없음)
*   **`style`**: UI/스타일 수정 (CSS 변경, 포맷팅 등)
*   **`docs`**: 문서 수정 (README, 주석, 가이드 문서 등)
*   **`chore`**: 기타 설정 변경 (패키지 매니저 설정, 빌드 업무 등)

**예시:**
*   `feat: 냉장고 식재료 추가 UI 구현`
*   `fix: 로그인 시 500 에러 발생하는 버그 수정`
*   `style: 버튼 색상 및 여백 조정`

---

## 2. Pull Request (PR) 가이드

작업을 완료하고 브랜치를 `dev` 또는 `main`으로 병합하기 위해 PR을 생성할 때, 아래의 템플릿을 사용하여 내용을 작성해 주세요.

### 📌 PR 제목
**제목은 반드시 커밋 메시지와 동일하게 작성합니다.**
*   예: `feat: 냉장고 식재료 추가 UI 구현`

### 📌 PR 본문 템플릿
PR 본문에 아래 내용을 복사하여 붙여넣고, 본인의 작업에 맞게 내용을 채워주세요.

```markdown
### 📝 작업 내용
- (무엇을 작업했는지 간략히 적어주세요)

### 🔄 변경 사항
- (어떤 코드/파일이 어떻게 변경되었는지 구체적으로 적어주세요)
- (예: app/routes/fridge.py 라우터 구조 변경)

### ✅ 테스트 / 확인 사항
- [ ] 로컬 실행 확인
- [ ] 주요 기능 동작 확인
- [ ] dev 브랜치 최신 내용 반영 확인

### 💡 참고 사항
- (리뷰어가 알아야 할 특이사항이나 발생했던 문제, 남은 이슈 등이 있다면 적어주세요. 없으면 생략 가능)
```

---

## 3. 작업 프로세스 권장 사항

1.  **브랜치 최신화**: 작업 시작 전 항상 `dev` 브랜치의 최신 변경 사항을 본인의 로컬 브랜치로 가져와주세요 (`git pull origin dev`).
2.  **독립적인 작업**: 충돌 방지를 위해 본인이 담당한 파일이나 기능(예: `fridge` 관련 파일)만 수정하는 것을 원칙으로 합니다.
3.  **잦은 커밋**: 의미 있는 단위로 잘게 나누어 커밋해주시면, 문제 발생 시 롤백이 쉬워집니다.

위 가이드라인을 잘 준수하여 안정적인 프로젝트 통합을 이루어 냅시다. 감사합니다!
