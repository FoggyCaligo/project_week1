# 🚀 프론트엔드/백엔드 통합 로컬 테스트 가이드 (팀원 전체용)

이 문서는 팀원 분들이 최신 통합 브랜치를 자신의 로컬(PC) 환경에서 실행하고, 정상적으로 작동하는지 테스트하기 위한 **통합 설명서**입니다.
**팀원2 (냉장고 파트)**의 블루프린트 구조 변환, DB 모델/서비스 분리, 공공데이터 API 연동 사항이 모두 반영되어 있습니다.

---

## 1. 프로젝트 초기 세팅

### 1-1. 레포지토리 풀 받기 및 가상환경 설정
터미널(또는 CMD/PowerShell)을 열고, 최신 변경사항을 가져옵니다.

```bash
# 최신 코드 반영 (dev 등 통합 브랜치 기준)
git pull origin dev

# 가상환경 생성 (최초 1회)
python -m venv venv

# 가상환경 활성화
# Windows (PowerShell):
.\venv\Scripts\activate
# Mac/Linux:
source venv/bin/activate
```

### 1-2. 필수 라이브러리 설치
새롭게 추가된 `flask-sqlalchemy`, `python-dotenv`, `requests` 등의 패키지를 설치합니다.

```bash
pip install -r requirements.txt
```

---

## 2. 환경 변수 및 데이터베이스 설정

### 2-1. `.env` 파일 생성
보안을 위해 DB 접속 정보나 비밀키 등은 코드에 올리지 않습니다. 프로젝트 최상위 경로(`app.py`나 `run.py`가 있는 폴더)에 `.env` 파일을 만들고 아래 내용을 복사해 붙여넣습니다.

```env
# 데이터베이스 접속 정보 (클라우드 DB)
DB_HOST=43.201.47.181
DB_PORT=3306
DB_USER=kool
DB_PASSWORD=master
DB_NAME=cloud33

# Flask 보안 키 (세션 등에 사용됨)
SECRET_KEY=today-fridge-secret-key
```
*(주의: 이 파일은 `.gitignore`에 등록되어 깃허브에 올라가지 않습니다. 로컬에서 각자 만들어야 합니다.)*

### 2-2. 데이터베이스 테이블 초기화 (테스트용)
`Flask-SQLAlchemy`를 통해 `user_ingredients` 등의 테이블을 DB에 자동 생성합니다. 아래 명령어를 실행하세요.

```bash
python init_db.py
```
> 출력결과: `Database tables created successfully!` 메시지가 나오면 성공입니다.

## 3. 웹 서버 실행

모든 준비가 끝났습니다! 아래 명령어로 로컬 서버를 가동합니다.

```bash
python run.py
```

- 터미널에 `Running on http://127.0.0.1:5000/` 문구가 출력되면 성공입니다.
- 웹 브라우저를 열고 **`http://127.0.0.1:5000/`** 로 접속하세요.

## 4. 🔍 통합 테스트 시나리오 (팀원 2 냉장고 파트)

팀원 2가 담당한 냉장고 파트의 코드가 제대로 동작하는지 아래 순서대로 테스트해 보세요.

1. **접속 테스트**
   - 로그인 여부 확인이 연동되어 있습니다. 테스트를 위해 브라우저 주소창에 `http://127.0.0.1:5000/fridge` 를 입력해 보세요.
   - UI 페이지가 정상적으로 렌더링되며 "❄️ 내 냉장고 식재료 관리" 가 뜨면 화면 연동이 성공한 것입니다. (현재는 강제 1번 유저로 세팅해두었습니다)
2. **기능 테스트 1: 식재료 추가**
   - 폼에 **재료명**(예: 감자), **분류**(채소), **유통기한**(내일 날짜)을 입력하고 `[추가]` 버튼을 누릅니다.
   - 화면 하단 리스트에 항목이 추가되고, DB에 저장되는지 확인합니다.
3. **기능 테스트 2: D-Day 뱃지(UI)**
   - 목록에 추가된 재료 옆에 남은 기간에 따라 D-Day 뱃지 색상이 다르게 뜨는지 확인합니다. (3일 이하는 노란색, 당일 및 경과는 빨간색, 그 외 초록색)
4. **기능 테스트 3: 공공데이터 레시피 자동 추천 연동**
   - 유통기한이 가장 짧은 식재료 리스트 아래에 **"💡 이런 레시피는 어때요? (임박 재료 활용)"** 영역이 생겼는지 확인합니다.
   - 실제 공공데이터 API(COOKRCP01)를 호출해, 냉장고 안의 재료로 만들 수 있는 레시피 정보(사진, 메뉴명 등)를 받아오는지 확인합니다.
5. **기능 테스트 4: 삭제**
   - 항목 우측의 `[삭제]` 버튼을 눌러 리스트와 DB에서 정상적으로 지워지는지 확인합니다.

## 5. 🛠 구조 변경점 안내 (필독)

팀의 컨벤션(패턴) 합의에 따라, 백엔드 코드가 아래와 같이 구조화(분리) 되었습니다. 다른 파트(추천, 게시판 등)를 개발하실 때도 이 구조를 참고해 주세요.

- `app/models/` (Entity): DB 테이블 구조를 정의합니다. (`ingredient.py` 참고)
- `app/services/` (Service): DB 연산(추가/삭제) 및 외부 API 연동 등 비즈니스 로직을 수행합니다. (`fridge_service.py` 참고)
- `app/routes/` (Controller): 브라우저 화면을 보여주거나 클라이언트의 요청(POST/GET)을 처리하고 데이터를 넘겨줍니다. (`fridge_views.py`, `fridge.py` 참고)
- 전체 라우팅은 **Blueprint**로 묶여 `app/__init__.py` 에 등록됩니다. 기존의 모놀리식 `app.py`는 더 이상 실행에 관여하지 않습니다.
